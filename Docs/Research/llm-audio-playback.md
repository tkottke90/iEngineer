# LLM Audio Playback Research

## Overview

This document covers how the AI engineer agent can deliver spoken audio to the driver's headset on a Windows PC in real time while racing. The architecture requires: the AI backend (server-side) generating audio bytes, and a mechanism to get those bytes played out through the driver's headset with minimal latency and friction.

**Key constraints driving this research:**

- Audio must reach a Windows PC and play through a headset
- TTS is out of scope here — this is purely about *delivery* of pre-generated audio
- Microphone capture (for driver voice input) is a bonus, not a hard requirement
- Free and open source solutions are strongly preferred

Two broad approaches emerged: **relay via an existing voice platform** (Discord, Mumble, TeamSpeak) and **direct delivery to a local client** (WebSocket/TCP + a small Python process on the Windows PC, or a media player with IPC control).

---

## Recommendation

### Primary: Discord Bot (Voice Channel Audio)

The recommended approach is a Discord bot that joins a private voice channel and plays audio generated by the AI backend. The driver is already in that voice channel through the Discord desktop app on their Windows PC, so the audio is delivered directly to their headset via Discord's existing voice infrastructure.

**Why this is the best fit:**

Most iRacing drivers already run Discord for team communication, so no additional client software is needed on the Windows side. Discord handles all the Windows audio routing, device selection, and headset integration through the app the driver is already using. The bot can also receive the driver's microphone audio from the same voice channel, satisfying the input requirement if needed down the line.

On the server side, a Discord bot created through the official Discord Developer Portal (not a "selfbot" — those are ToS violations) can be written in Python using `discord.py`. The bot joins a voice channel, and plays audio using `discord.FFmpegPCMAudio` to stream a file, or a custom `AudioSource` subclass to stream raw PCM bytes directly from the AI pipeline. The only system dependency is FFmpeg, which must be installed on the server running the bot.

**Architecture:**
```
AI Backend (server)
  → generates TTS audio bytes
  → Discord bot (discord.py + FFmpeg)
  → Discord voice infrastructure
  → Discord desktop app on Windows
  → driver's headset
```

**Tradeoffs:**
- Adds a dependency on Discord's infrastructure (internet required, Discord servers as relay)
- Introduces ~50–150ms of additional latency through Discord's voice routing vs. a direct connection
- Requires FFmpeg installed on the server
- Requires creating a Discord application and bot token (free, ~5 minutes)
- Microphone input from the voice channel requires the `discord-ext-voice-recv` extension (community-maintained)

---

### Alternative: WebSocket + Local Python Client

If Discord is undesirable (e.g., you want offline/LAN-only operation, lower latency, or no third-party dependency), a lightweight Python script running as a background process on the Windows PC is the simplest pure technical solution.

The AI backend sends audio bytes over a WebSocket connection. The local Python script receives each chunk and plays it through the selected audio device (headset) using `sounddevice` or `pyaudio`. The same script can simultaneously capture microphone input and stream it back to the server.

**Architecture:**
```
AI Backend (server)
  → WebSocket server
  → local Python client (Windows, background process)
  → sounddevice / pyaudio → headset
```

This approach has the lowest latency and fewest dependencies, but requires the driver to have Python installed and a small client script running on their gaming PC. For a personal project this is trivial; for distributing to others it adds friction.

## Options Evaluated

### 1. Discord Bot (discord.py)

**How it works:** A bot account created via the Discord Developer Portal joins a voice channel using `discord.py`. Audio is played using `VoiceClient.play()` with `FFmpegPCMAudio` (for file/pipe input) or a custom `AudioSource` subclass (for raw PCM byte streams from the AI pipeline). For voice receive, the community extension `discord-ext-voice-recv` mirrors the send API.

**Cost:** Free. Discord accounts and bots are free. The server hosting the bot needs FFmpeg.

**Latency:** Moderate. Discord routes voice through its own infrastructure, adding ~50–150ms over a direct connection. Acceptable for race coaching use cases where sub-100ms precision is not required.

**Mic input support:** Yes, via `discord-ext-voice-recv`. Audio received is 20ms Opus frames.

**TTS capability:** None built-in. The bot plays audio you give it.

**Complexity:** Low-to-medium. Creating a bot token and writing 50–80 lines of Python is straightforward. FFmpeg is the only non-Python dependency.

**Key libraries:**
- `discord.py` — [https://github.com/Rapptz/discord.py](https://github.com/Rapptz/discord.py)
- `discord-ext-voice-recv` — [https://github.com/imayhaveborkedit/discord-ext-voice-recv](https://github.com/imayhaveborkedit/discord-ext-voice-recv)
- Pycord (discord.py fork) — [https://docs.pycord.dev](https://docs.pycord.dev) (alternative with more voice features)

**Important note:** Discord prohibits "selfbots" — automating a normal user account. A proper bot account created through the Developer Portal is explicitly allowed and is the correct approach.

---

### 2. WebSocket + Local Python Client (sounddevice / pyaudio)

**How it works:** A Python WebSocket server on the AI backend streams audio chunks to a small Python client script running on the Windows PC. The client decodes and plays the audio through the headset using `sounddevice` (preferred — cleaner API, returns immediately, plays in background) or `pyaudio` (lower-level, PortAudio-based).

**Cost:** Free. All open source.

**Latency:** Very low — essentially just network latency on LAN, or internet RTT for remote deployment.

**Mic input support:** Yes. `sounddevice` and `pyaudio` both support recording. The client can open a bidirectional stream simultaneously.

**TTS capability:** None. Client just plays bytes it receives.

**Complexity:** Low. ~50–100 lines of Python on each side. The only friction is distributing and running a Python script on the Windows gaming PC.

**Key libraries:**
- `sounddevice` — [https://python-sounddevice.readthedocs.io](https://python-sounddevice.readthedocs.io) / [https://pypi.org/project/sounddevice/](https://pypi.org/project/sounddevice/)
- `pyaudio` — [https://pypi.org/project/PyAudio/](https://pypi.org/project/PyAudio/)
- `websockets` — [https://pypi.org/project/websockets/](https://pypi.org/project/websockets/)
- Reference: TCP socket audio streaming example — [https://github.com/richardos/audio-stream](https://github.com/richardos/audio-stream)

### 3. Mumble + pymumble Bot

**How it works:** Mumble is open-source, self-hosted voice chat software. `pymumble` is a Python client library that connects to a Murmur (Mumble server) instance and can both send and receive audio. The AI bot connects as a client, and the driver hears it through the Mumble desktop app.

**Cost:** Free / open source (GPL-3.0). Self-hosted Murmur server required.

**Latency:** Low — Mumble is specifically designed for low-latency voice.

**Mic input support:** Yes — pymumble can receive audio from other users in the channel.

**TTS capability:** None built-in.

**Complexity:** Medium. Requires running a Murmur server. Requires the driver to install and configure Mumble.

**Major concern:** The primary `pymumble` fork (`azlux/pymumble`) is effectively unmaintained. The maintainer posted a notice in the README stating they no longer use Mumble, lack motivation to maintain the library, and that it has architectural problems — specifically that it is not asynchronous, causing latency to increase with connection uptime. Last release was May 2021. The Mumble team is apparently working on a replacement library, but it is not yet available.

**Key libraries:**
- `pymumble` (azlux fork) — [https://github.com/azlux/pymumble](https://github.com/azlux/pymumble)
- Mumble — [https://github.com/mumble-voip/mumble](https://github.com/mumble-voip/mumble)
- Example bot (music): `botamusique` — [https://github.com/azlux/botamusique](https://github.com/azlux/botamusique)

### 4. LiveKit (WebRTC, self-hostable)

**How it works:** LiveKit is an open-source WebRTC SFU (Selective Forwarding Unit) platform designed for real-time audio/video. It has a Python Agents SDK (`livekit-agents`) specifically designed for connecting AI pipelines to voice rooms. The driver connects to the LiveKit room through a browser or native app; the AI agent joins the same room and publishes audio.

**Cost:** Free / open source (Apache 2.0). LiveKit Cloud offers a hosted tier. Self-hosting is via Docker or Kubernetes.

**Latency:** Very low (WebRTC, ~20–50ms).

**Mic input support:** Yes — full bidirectional audio via WebRTC.

**TTS capability:** The Agents SDK has plugin integrations for many TTS providers, but core delivery is TTS-agnostic.

**Complexity:** High for self-hosting. Setting up a LiveKit server, configuring STUN/TURN, and building a client for the Windows PC is a significant undertaking compared to other options here. Most appropriate if this project eventually needs multi-user voice rooms or cloud-scale deployment.

**Key libraries:**
- LiveKit server — [https://github.com/livekit/livekit](https://github.com/livekit/livekit)
- Python SDKs — [https://github.com/livekit/python-sdks](https://github.com/livekit/python-sdks)
- Agents framework — [https://github.com/livekit/agents](https://github.com/livekit/agents)

---

### 5. aiortc (WebRTC, Python)

**How it works:** `aiortc` is a pure Python WebRTC / ORTC library built on asyncio. It can act as a WebRTC peer and stream audio tracks to a browser or other WebRTC client. The Windows client would be a browser tab or Electron app connecting to the aiortc server.

**Cost:** Free / open source (BSD).

**Latency:** Very low (~20ms Opus frames).

**Mic input support:** Yes — the browser client can send microphone audio back.

**TTS capability:** None.

**Complexity:** Medium-high. WebRTC signaling, ICE negotiation, and STUN/TURN configuration add significant complexity for what is a relatively simple use case. Using a browser as the "client" on the Windows PC is possible but awkward for a gaming overlay.

**Key libraries:**
- `aiortc` — [https://github.com/aiortc/aiortc](https://github.com/aiortc/aiortc)

---

### 6. mpv with IPC Control

**How it works:** `mpv` is an open-source media player with a JSON IPC interface over named pipes (Windows) or Unix sockets (Linux/macOS). A script on the AI backend writes audio files and sends `loadfile` commands to a running `mpv` process on the Windows PC, which plays the audio through the configured audio device.

**Cost:** Free / open source.

**Latency:** Moderate. Involves writing to disk, issuing IPC commands, and waiting for the player to load and play the file. Not designed for streaming audio bytes in real time.

**Mic input support:** No — mpv is a player only.

**TTS capability:** None.

**Complexity:** Low-to-medium for basic playback, but the IPC protocol on Windows (named pipes) is awkward to work with from Python and has documented reliability issues with multiple sequential commands.

**Key resources:**
- mpv IPC documentation — [https://github.com/mpv-player/mpv/blob/master/DOCS/man/ipc.rst](https://github.com/mpv-player/mpv/blob/master/DOCS/man/ipc.rst)

---

### 7. VLC HTTP Audio Streaming

**How it works:** VLC can serve an HTTP audio stream that another VLC instance (or any media player) on the Windows PC consumes. The AI backend writes audio to a file or pipe that VLC transcodes and streams.

**Cost:** Free / open source.

**Latency:** High for streaming scenarios. HTTP-based media streaming is designed for throughput, not latency. Buffering adds multiple seconds of delay.

**Mic input support:** No.

**TTS capability:** None.

**Complexity:** Medium, but latency characteristics make it unsuitable for real-time race coaching.

---

### 8. TeamSpeak Bot

**How it works:** Similar conceptually to Discord/Mumble — a bot connects to a TeamSpeak server and plays audio. The TeamSpeak SDK uses a C interface, and Python integrations are limited and fragile. Some community Python bots exist but are experimental.

**Cost:** Free for personal/non-commercial use. Server-side SDK licensing can be complex for commercial use.

**Latency:** Low.

**Mic input support:** Via SDK.

**Complexity:** High. The C SDK with Python bindings is the official route, and the ecosystem for audio-injecting Python bots is thin. Not worth the complexity given better alternatives.

---

## Summary Comparison

| Option | Cost | Complexity | Latency | Mic Input | Client Software Required |
|---|---|---|---|---|---|
| **Discord Bot** | Free | Low | Moderate | Yes (ext) | Discord app (likely already installed) |
| **WebSocket + Python Client** | Free | Low | Very low | Yes | Python + script on Windows PC |
| **Mumble + pymumble** | Free | Medium | Low | Yes | Mumble app; library unmaintained |
| **LiveKit** | Free/Cloud | High | Very low | Yes | Browser or native app |
| **aiortc** | Free | Medium-High | Very low | Yes | Browser tab |
| **mpv IPC** | Free | Medium | Moderate | No | mpv player |
| **VLC HTTP** | Free | Medium | High | No | VLC player |
| **TeamSpeak Bot** | Free* | High | Low | Yes | TeamSpeak app |
