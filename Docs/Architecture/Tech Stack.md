# Tech Stack

## Overview

The system is composed of two primary runtime environments: a lightweight **Tauri client** that runs alongside iRacing on the racing PC, and a **Node.js hub server** that owns integrations, inference, and the primary product UI. Redis connects them as the shared data bus.

---

## Monorepo Structure

**Tooling:** npm Workspaces

The repository is a monorepo with the following top-level packages:

- `apps/tauri-client` — Tauri 2 app (Rust + Preact)
- `apps/hub-server` — Node.js hub server (hono-preact)
- `packages/ui` — Shared Preact component library
- `packages/types` — Shared TypeScript types (telemetry, race state, events) — consumed by hub server and web UI

npm Workspaces handles inter-package linking and script orchestration. No additional build orchestration layer (Turborepo, Nx) at this time.

### Cross-Language Type Safety

The Tauri client is Rust and cannot consume the TypeScript `packages/types` package directly. Two tools maintain type contract across the language boundary:

**`ts-rs`** (Redis message boundary) — Rust structs that cross the Redis boundary (telemetry ticks, race state events, voice commands) are annotated with `#[derive(TS)]`. Running `cargo test` in `apps/tauri-client` exports `.ts` declaration files into `packages/types`. The generated TypeScript matches `serde_json` serialization exactly, including serde rename and tag attributes. This is the source of truth for the Redis message schema — Rust defines it, TypeScript consumes the export.

**`tauri-specta`** (Tauri IPC boundary) — Tauri `#[command]` functions exposed to the Preact webview get typed wrappers generated by tauri-specta. This covers the in-process IPC between the Tauri Rust backend and the Tauri webview frontend, which is a separate boundary from Redis.

Both tools coexist without conflict and are the standard pattern in the Tauri v2 community.

## Tauri Client (Racing PC)

**Framework:** Tauri 2 (Rust + Preact)

The Tauri app is intentionally minimal. It runs on the same machine as iRacing and is kept lightweight to avoid competing for resources during a session. Its UI is limited to settings and a debug/diagnostics view.

### Responsibilities

1. **Telemetry reading** — Reads the iRacing shared memory interface at 60 Hz via a custom Rust integration. No community SDK crate; the implementation is written against the official `irsdk_defines.h` specification so we own it entirely.
2. **Telemetry publishing** — Publishes telemetry ticks to Redis Streams for consumption by the Node.js server.
3. **Voice I/O** — Captures microphone audio via Rust (`cpal`), resamples to 16kHz mono PCM, and sends the buffered clip via HTTP POST to Speaches (on the homelab server) on PTT key-release. Receives an MP3 audio URL from the hub server via Redis Pub/Sub, fetches the clip over LAN, and plays it back via Rust (`rodio`).
4. **Hotkey triggers** — Listens for global hotkeys (and Stream Deck bindings) and publishes command events to Redis.
5. **iRacing camera control** — Receives camera switch commands from the Node.js server and executes them via iRacing SDK broadcast messages.

### Local Configuration Storage

**Database:** SQLite (via Tauri's built-in SQLite plugin)

All local configuration is persisted in a SQLite database on the racing PC:

- Audio device selection
- Connection settings (Redis URL, Whisper endpoint, hub server URL)
- Hotkey bindings
- Opt-in/opt-out telemetry logging preference

SQLite is appropriate here: the data is small, structured, and local to a single machine. No server or network dependency.

### Tauri UI (minimal)

**Frontend:** Preact + preact-iso (SPA, no SSR — runs in the local Tauri webview)

- Audio device selection
- Redis, Whisper, and hub server connection configuration
- Hotkey bindings
- Telemetry logging opt-in toggle
- Live telemetry debug readout (variable values, connection status)

## Node.js Hub Server

**Framework:** Node.js + [hono-preact](https://github.com/sbesh91/hono-preact) (`nodeAdapter`)

The Node.js server is the product brain. It consumes telemetry from Redis, runs strategy logic, calls the inference service, and drives all integrations. It also serves the primary web UI.

hono-preact is a lightweight full-stack framework pairing Hono (server) with Preact (browser). It provides SSR with client-side hydration, typed server loaders and actions, streaming, and WebSocket support. The existing Hono routes (webhooks, health checks, API endpoints) compose cleanly alongside it via `src/api.ts`.

**Deployment:** Docker image. The hub server is packaged and run as a Docker container on the homelab server, co-located with the PostgreSQL and Redis containers via Docker Compose.

### Responsibilities

- **Telemetry consumption** — Reads from Redis Streams, builds and maintains live race state
- **Strategy engine** — Processes race state, triggers proactive engineer briefings
- **LLM inference** — Calls the configured inference endpoint with race context
- **TTS dispatch** — Sends LLM responses to the local TTS service, returns audio to Tauri for playback
- **OBS controller** — WebSocket client to OBS Studio (obs-websocket v5); executes scene switches, source visibility changes, replay buffer saves
- **Overlay server** — HTTP server hosting overlay HTML templates consumed by OBS browser sources
- **Discord bridge** — Posts stint summaries, pit alerts, and swap recommendations to team Discord channels via webhooks
- **Broadcast plan engine** — Evaluates broadcast plan rules against live race state to drive autonomous stream decisions
- **Web UI server** — Serves the primary product interface (see below)
- **iRacing REST API client** — Fetches post-session lap data, driver stats, and historical context

### Primary Web UI

**Frontend:** Preact via hono-preact (SSR + client-side hydration, streaming loaders, WebSocket-capable)

The main product interface is a web application served by the Node.js server. This makes it accessible from any device on the LAN — the driver's PC, a co-pilot's laptop, or a tablet used by the Stream Operator.

**Screens:**

- Broadcast plan editor (pre-session)
- Race control center / stream operator view (during session)
- Post-session debrief
- Team coordination dashboard

#### Real-Time Updates (Race Control Center)

The race control center receives live race state via **WebSocket** connection to the hub server. This is the same WebSocket infrastructure already present in hono-preact/Hono, and aligns with the OBS WebSocket integration already in use.

#### Video Feed in Race Control Center

The OBS WebSocket protocol includes a video feed capability, but it is limited to ~1 FPS — too slow for a usable stream preview. The race control center video panel instead consumes an **HLS stream** served independently of the hub server.

The HLS stream URL is operator-configured. How the HLS stream is produced is outside the application's scope — common sources include OBS's built-in HLS output, a media relay receiving an RTMP ingest, or a streaming helper application. The race control page embeds the URL in a standard `<video>` element (with HLS.js for cross-browser support). No video data passes through the hub server.

### Shared Component Library

Both the Tauri UI and the Web UI use Preact, enabling a shared `packages/ui` workspace with common components, hooks, and types. The monorepo structure mirrors the one hono-preact itself uses.

## Data Layer

### Redis (always running, including solo use)

Redis is the shared data bus between the Tauri client and the Node.js server. It is used even in solo (single-driver) sessions to maintain a single consistent architecture rather than a conditional solo-vs-team code path.

- **Redis Streams** — High-frequency telemetry ingestion from one or more Tauri clients
- **Pub/Sub** — Event broadcasting (camera commands, voice triggers, flag changes)
- **Key/Value** — Current race state snapshot, configuration

In team sessions, multiple Tauri clients publish to the same Redis instance. The Node.js server merges their data with driver-priority rules (active driver's telemetry takes precedence over observer data for player-only variables).

#### Hosting

Redis runs as a Docker container on the homelab server, defined in the same Docker Compose stack as the hub server and PostgreSQL.

**Access model:** The homelab operator controls access via network-level rules (firewall, Redis `AUTH` password, or VPN). Teammates connect their Tauri clients to the homelab Redis — no app-level account system is needed for v1.

**Resilience:** The Tauri client handles Redis unavailability gracefully. If the homelab server goes down mid-race, the race session continues normally — the engineering layer (voice responses, stream control) goes quiet, but nothing crashes.

### PostgreSQL (Server-Side Persistence)

**Hosting:** Docker container, co-located with the hub server in the Docker Compose stack.

PostgreSQL stores server-side application data that must persist across sessions and server restarts:

- User accounts and preferences (linked to Authentik identities)
- Broadcast plan definitions and history
- Team configuration (driver roster, swap windows, stint assignments)
- Post-session race summaries and debrief data cache
- Application event log (strategy decisions, engineer briefings, scene switches)

### iRacing REST API (post-session)

Historical and reference data fetched on demand — lap charts, session results, driver stats, car and track metadata. Not used in the real-time path.

Post-session data is fetched from the REST API and persisted to PostgreSQL for the current debrief session, then available for historical review.

## Authentication

**Provider:** Authentik (homelab-hosted identity provider)
**Protocol:** OAuth2 / OIDC

The hub server's web UI is protected via Authentik, the homelab's existing identity provider. Users authenticate through the standard OAuth2 authorization code flow. This reuses an existing service rather than introducing app-level credential management.

The Tauri client uses a pre-shared connection token (configured per-installation) to authenticate with the hub server API and Redis — no full OAuth flow in the native app.

---

## Overlay System

**Model:** HTML templates with JavaScript API integration

The application's responsibility is:

1. Hosting the overlay HTML pages at stable local URLs
2. Exposing a WebSocket or REST API that overlays can query for live race state

The overlays themselves are self-contained HTML/CSS/JS files that pull data from the app's API on a timer or via WebSocket subscription. Users can author overlays in any style — DOM-based, Canvas-based, or SVG — as long as they use the data API.

This decouples the overlay rendering engine from the application core. The app ships a set of default overlay templates; advanced users can fork and customize them.

OBS browser sources load the overlay URLs directly. No data is pushed from the app into OBS for overlays — OBS polls the URL, and the page's own JS handles data refresh.

## Logging & Observability

**Standard:** OpenTelemetry (OTel) across all components. **Grafana Alloy** runs on the homelab server as the OTel collector, receiving signals from all services and routing them to the appropriate Grafana backend:

| Signal         | Backend            | Purpose                                                             |
| -------------- | ------------------ | ------------------------------------------------------------------- |
| Logs           | Grafana Loki       | Structured application logs, searchable by service and session      |
| Traces / Spans | Grafana Tempo      | Distributed traces — including the audio pipeline latency waterfall |
| Metrics        | Grafana Prometheus | Optional; available for throughput and error rate dashboards        |

All backends are homelab-hosted and Grafana-native, so they compose into a single Grafana dashboard without additional configuration.

### Tauri App

The Tauri client generates structured logs locally and ships them to the hub server's log ingestion endpoint. Shipping is subject to an opt-in "allow telemetry" toggle stored in the local SQLite config. If the user opts out, logs remain on the local machine only.

The hub server receives Tauri logs and forwards them to Alloy, tagged with the originating machine and session ID, for routing to Loki.

### Hub Server (Web App & API)

Uses [tkottke90/js-libraries logger package](https://github.com/tkottke90/js-libraries/tree/main/packages/logger) as the logging foundation. The logger is configured to emit to:

- **stdout** — structured JSON, captured by Docker and available via `docker logs`
- **Alloy** — via OpenTelemetry OTLP exporter; Alloy routes logs to Loki and spans to Tempo

### Distributed Tracing

The audio pipeline latency waterfall (defined in the Audio Pipeline section) emits OTel spans that flow to Grafana Tempo. A trace ID scoped to each voice interaction correlates all stages — PTT capture through audio playback — into a single Tempo trace, giving a flamegraph view of where time is spent.

All other session-scoped events are tagged with a `session_id` for log correlation in Loki.

### Log Levels

Log levels follow standard conventions (`debug`, `info`, `warn`, `error`). Debug-level telemetry logging is off by default to avoid volume at 60 Hz.

## Speech Services

Both speech services run as persistent local processes on the homelab server. Neither depends on cloud services.

| Service                                 | Role           | Host           | Notes                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------------------------------- | -------------- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Whisper** (via Speaches)              | Speech-to-text | Homelab server | [Speaches](https://github.com/speaches-ai/speaches) (`speaches-ai/speaches`) running the `base.en` model. Tauri sends a 16kHz mono PCM audio file via HTTP POST on PTT release and receives the transcript. LAN audio transfer adds ~1–10ms — negligible against ~200ms inference time. Co-located with the hub server in Docker Compose.                                                      |
| **Chatterbox** (via chatterbox-tts-api) | Text-to-speech | Homelab server | [travisvn/chatterbox-tts-api](https://github.com/travisvn/chatterbox-tts-api) — a FastAPI wrapper exposing an OpenAI-compatible `/v1/audio/speech` endpoint with named voice profile management. Co-located with the hub server in Docker Compose. Target output latency under 300ms. See Chatterbox Configuration section below for model variant, voice cloning, and expression tag details. |

### Chatterbox Configuration

#### Model Variant

Two Chatterbox model variants are supported, selected per use case via configuration:

| Variant             | Parameters | Decoder steps | Paralinguistic tags | Tuning controls                                       | Use case                                                    |
| ------------------- | ---------- | ------------- | ------------------- | ----------------------------------------------------- | ----------------------------------------------------------- |
| **Turbo** (default) | 350M       | 1-step        | ✅ 9 confirmed tags | `temperature`, `top_p`, `top_k`, `repetition_penalty` | Real-time race engineer voice responses                     |
| **Original**        | 500M       | 10-step       | ❌                  | `exaggeration`, `cfg_weight`, `temperature`           | Pre/post-session analysis narration, non-real-time contexts |

Turbo is the default for all real-time voice output. The Original variant is available for contexts where latency is not critical and where `exaggeration` (prosodic intensity) or `cfg_weight` (voice adherence) control is useful — for example, a post-session debrief summary rendered to an audio file. Both models cannot run simultaneously without loading both into VRAM; the active variant is selected per request.

#### Voice Profiles

The app supports two voice sources, selectable per installation:

**Default voices** — The app ships with one or more pre-registered voice audio files bundled into the Chatterbox Docker volume. No user setup required. These are the out-of-the-box engineer voices, referenced by name in all TTS calls.

**User-cloned voices** — Users upload a reference audio clip (WAV, 16kHz+, ~10 seconds, single speaker, clean audio) via the **web UI settings screen**. The hub server forwards the file to the Chatterbox API (`POST /v1/voices` with `voice_name` + `voice_file`), which stores it on the Docker volume and makes it available for subsequent TTS requests by name. No audio bytes are sent with each TTS request — only the voice name string. The Chatterbox server re-runs conditioning on the stored audio file at synthesis time.

Voice profile management (list, upload, delete) is handled entirely through the web UI. No recording UI is provided in the Tauri app — users supply a pre-recorded audio file. The web UI displays quality guidance (quiet environment, single speaker, ~10 seconds) alongside the upload control.

#### Paralinguistic Expression Tags (Turbo only)

When using the Turbo variant, the LLM system prompt includes instructions to use expression tags where contextually appropriate. Tags are embedded inline in the generated text and Chatterbox Turbo renders them as non-verbal sounds rather than spoken words.

Supported tags available to the LLM:

`[cough]`, `[clear throat]`, `[sigh]`, `[chuckle]`, `[laugh]`, `[groan]`, `[sniff]`, `[gasp]`, `[shush]`

The LLM decides when and whether to include tags. This approach accepts the risk of occasional overuse; if that becomes an issue, tag instructions can be removed from the system prompt without any other changes.

#### MP3 Output and FFmpeg

Chatterbox outputs a raw audio tensor (WAV at 24kHz). MP3 conversion is handled by an FFmpeg process in the Docker Compose stack. The conversion step adds ~10–30ms of latency and is negligible against TTS inference time. The Resemble AI PerTh watermark embedded in all Chatterbox output survives MP3 compression intact.

## Inference Service

The hub server calls an LLM via a standard API. Two deployment modes are supported and toggled via configuration — no code changes required to switch:

| Mode         | Provider                                                                          | Use Case                                                    |
| ------------ | --------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **Frontier** | Anthropic Claude API (claude-3.5-sonnet or later)                                 | High-quality responses; requires internet and API key       |
| **Local**    | GLM family via OpenAI-compatible local inference server (Ollama, LM Studio, vLLM) | No cloud dependency; latency and quality depend on hardware |

The hub server targets the OpenAI chat completions API shape in both cases (Anthropic's SDK handles the translation for Claude). Model name, base URL, and API key are all configurable at runtime. The server degrades gracefully if the inference service is unreachable.

Model selection, quantization level, and hardware sizing for the local path are deferred — these are operational tuning decisions, not application architecture decisions.

## Audio Pipeline

The voice interface has two legs: voice-in (mic → Whisper → text → hub) and voice-out (hub → Chatterbox → audio → playback). Inference (Whisper, Chatterbox) runs on the homelab server. Audio I/O — mic capture and speaker playback — is owned by the Rust layer in the Tauri client via `cpal`/`rodio`, keeping it at the system level and out of the Web Audio API.

### Leg 1: Voice In

**Activation model: Push-to-Talk (PTT)**

The driver holds a configured hotkey while speaking. PTT bounds the utterance cleanly — no voice activity detection needed — and matches real-world radio discipline. The Tauri client starts buffering audio on key-down and stops on key-release.

**Mic capture: Rust via `cpal`**

Audio capture is owned by the Rust layer using the `cpal` crate (cross-platform audio I/O). The webview is not involved in capture.

**Resampling: Rust, before sending**

Microphones typically output at 44.1kHz or 48kHz stereo. Whisper expects 16kHz mono 16-bit PCM. The Tauri client resamples and converts the audio to the target format in Rust (via the `rubato` crate) before any network transfer. This keeps the Whisper server focused on transcription rather than format normalization.

**Format: 16kHz mono 16-bit PCM**

This is Whisper's native input format. No additional encoding step.

**Whisper server: Speaches on homelab**

Speaches runs as a Docker container on the homelab server, co-located with the hub server, Redis, and Chatterbox. Tauri sends audio to Speaches over LAN. The network hop adds ~1–10ms — negligible against ~200ms inference time. Running Whisper on the homelab rather than the racing PC avoids loading an ML model and competing for CPU with iRacing, OBS, and game overlays.

**Whisper model: base.en with initial_prompt**

`base.en` (74M parameters) is the starting model. At ~200ms inference time for a 3-second clip on CPU, it meets the latency budget. Accuracy is sufficient for short, constrained-vocabulary commands. Speaches supports an `initial_prompt` parameter that primes the model with expected vocabulary before each transcription — a prompt covering racing terms ("fuel level", "pit window", "gap to P2", "tire compound", "laps remaining") improves recognition of domain-specific phrases for both `base.en` and any larger model. Upgrade to `small.en` only if recognition failures are observed on noisy audio or variable phrasing.

**Whisper transport: HTTP POST on PTT release**

On PTT key-release, Tauri sends the buffered audio as a single HTTP POST to the Speaches `/v1/audio/transcriptions` endpoint. A 3-second clip at 16kHz mono 16-bit PCM is ~100KB — upload completes in under 10ms on LAN. Streaming upload during the PTT hold offers no meaningful latency improvement at this clip size and is not used.

**Transcription delivery to hub: Redis Pub/Sub**

Once Speaches returns the transcription, Tauri publishes it to a Redis Pub/Sub channel. The hub server subscribes and picks it up for LLM inference. This keeps the voice input path consistent with the rest of the event architecture.

---

### Leg 2: Voice Out

**TTS output format: MP3 via FFmpeg**

Chatterbox outputs a raw WAV tensor at 24kHz. MP3 conversion is handled by FFmpeg, which runs as part of the homelab Docker Compose stack. The conversion adds ~10–30ms of latency — negligible against TTS inference time. WAV is not used for delivery due to file size (~12× larger than MP3 for equivalent quality on typical voice output).

**Audio delivery: Hub publishes URL → Tauri fetches and streams**

1. Hub server generates the TTS audio clip
2. Hub stores the clip in an in-memory cache with a short TTL (60 seconds)
3. Hub exposes the clip at a local HTTP URL (a route on the hub server)
4. Hub publishes the URL to a Redis Pub/Sub channel that Tauri subscribes to
5. Tauri receives the URL, appends it to its local playback queue, and fetches/streams the audio

The hub never pushes binary audio data over Redis or WebSocket — it serves the audio as a standard HTTP response to a Tauri-initiated request. This keeps audio delivery simple and lets Tauri control when and how it fetches.

**In-memory audio cache**

Clips are held in memory on the hub server with a 60-second TTL. No disk writes for TTS output. Clips are evicted automatically after TTL. If Tauri misses the Pub/Sub message and the URL expires, the clip is lost — this is acceptable for real-time voice responses, which are only relevant in the moment.

**Tauri playback: Rust via `rodio`**

Playback is owned by the Rust layer using the `rodio` crate, which wraps `cpal` and handles MP3 decoding and audio output. The webview is not involved in playback.

**Playback queue: Tauri-managed in-process queue**

Tauri maintains its own in-process playback queue. When a new audio URL arrives via Redis Pub/Sub, it is appended to the queue. The queue plays clips sequentially. Redis Pub/Sub is the notification channel only — it does not act as the queue. This distinction matters: Pub/Sub messages are fire-and-forget, so if Tauri missed a message it would be lost. By queuing on the client side and fetching via HTTP, missed Pub/Sub messages can be handled with a retry or catchup mechanism if needed in future.

**Interruption**

Priority audio events (yellow flag, critical alerts) can preempt the current playback by clearing the queue and stopping the active `rodio` sink. A priority field on the Pub/Sub message signals when to interrupt vs. queue.

### Cross-Cutting: Latency Budget and Observability

**End-to-end target: under 5 seconds** from PTT release to first audio playback. This is the initial baseline — track and tighten it as the system matures.

**Per-stage OTel tracing**

Each leg of the pipeline emits an OpenTelemetry span, producing a waterfall view in Grafana that shows exactly which stage is the bottleneck:

| Stage                 | Start                   | End                       |
| --------------------- | ----------------------- | ------------------------- |
| PTT capture           | Key-down                | Key-release               |
| Whisper transcription | Upload start            | Transcription received    |
| Hub processing        | Transcript received     | LLM call dispatched       |
| LLM inference         | Call dispatched         | Full response received    |
| TTS generation        | Text sent to Chatterbox | Audio clip ready          |
| Audio delivery        | URL published           | Tauri fetch complete      |
| Playback start        | Fetch complete          | First audio sample played |

Span data flows into Grafana Loki/Tempo via the OpenTelemetry pipeline already used for application logs. A session-scoped trace ID correlates all stages of a single voice interaction.
