# Feature Specification: Tauri Client Configuration UI

**Feature Branch**: `006-client-config-ui`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "M10 — Tauri Client Configuration UI: driver can configure the app without editing config files"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Audio Device Configuration (Priority: P1)

A driver opens the app settings, selects their headset microphone as the PTT input device and their speaker as the engineer voice output, then saves. On the next race session, push-to-talk uses the correct mic and the engineer speaks through the correct output.

**Why this priority**: Audio routing is the most immediate barrier to a working session. A driver who can't hear the engineer or whose mic isn't captured gets no value from any other feature.

**Independent Test**: Can be fully tested without an active iRacing session by opening the settings page, selecting devices from dropdowns, saving, then triggering a sample playback clip through the selected output and verifying the mic level meter responds to voice input.

**Acceptance Scenarios**:

1. **Given** the settings page is open, **When** the driver opens the audio output dropdown, **Then** all available system audio output devices are listed by name.
2. **Given** the driver selects an output device and saves, **When** the engineer delivers the next voice message, **Then** audio plays through the selected device.
3. **Given** the driver selects an output device and saves, **Then** the application plays a short test clip through the selected output device without starting a session.
4. **Given** the driver selects a microphone input, **When** the driver speaks into it, **Then** the mic level meter in the settings page visibly responds.
5. **Given** a previously saved audio device is no longer available (unplugged), **When** the settings page opens, **Then** the device is shown as "unavailable" and the driver is prompted to reselect.

### User Story 2 — Connection Configuration (Priority: P1)

A driver sets up the app on a new machine or changes their homelab configuration. They open settings, enter the Redis URL, hub server URL, and LLM API base URL + model name, then confirm connectivity before starting a session.

**Why this priority**: Without correct connection settings, nothing works. On a fresh install or after a homelab change, the driver must be able to fix this without editing config files.

**Independent Test**: Can be tested by entering a valid hub URL, pressing the hub's "Test" button (tests are per-service — FR-008), and seeing a success indicator; and separately entering an invalid URL and seeing a clear failure message with the specific issue.

**Acceptance Scenarios**:

1. **Given** the settings page is open, **When** the driver enters connection details and presses each service's "Test" button (Test Redis / Test Hub / Test LLM — tests are per-service, per FR-008), **Then** that service reports connected or the specific reason for failure.
2. **Given** a saved configuration, **When** the app starts, **Then** it connects using the saved values without requiring re-entry.
3. **Given** the driver enters a malformed URL, **When** they attempt to save, **Then** the field shows an inline validation error before saving.
4. **Given** the LLM endpoint is configured, **When** the driver saves the model name, **Then** the Racing Engineer service uses that model for the next LLM synthesis request (whether user-initiated via PTT or auto-triggered) without requiring an app restart or hub server restart.
5. **Given** a fresh install (`first_launch_seen == false`), **When** the Connection tab opens, **Then** a dismissable hint ("Default — update for your setup") is shown adjacent to the pre-filled LLM base URL and model name fields; dismissing it hides it for the current session only. **When** any explicit Save-button save succeeds (whether or not the LLM fields were modified from their defaults; the internal PTT auto-save from `bind_ptt_hotkey()` does NOT count — A2), **Then** `first_launch_seen` is set to `true` and the hint does not appear on any subsequent launch; a failed Save does NOT suppress the hint. (B3 — tests the FR-007 first-launch hint lifecycle end-to-end.)

### User Story 3 — PTT Hotkey Binding (Priority: P2)

A driver assigns a key or button as their push-to-talk trigger. The binding works even when the sim window is focused. A Stream Deck user routes a button through the passthrough without special configuration.

**Why this priority**: PTT is core to M5's on-demand query flow. Without an easy way to bind and verify the key, the driver cannot use voice queries without editing files.

**Independent Test**: Can be tested by clicking "Set PTT Key," pressing a key on the keyboard (or a Stream Deck button), seeing the key name displayed, then speaking a query and verifying the transcript appears in the debug readout.

**Acceptance Scenarios**:

1. **Given** the driver clicks "Set PTT Key" and presses a key, **Then** the binding updates and displays the key name.
2. **Given** a PTT key is bound and iRacing is running in windowed or borderless-windowed mode, **When** the driver holds the PTT key with the sim window focused, **Then** the PTT-active indicator in the Hotkeys tab lights up for the duration of the hold (confirming the global shortcut fired without app focus). **A1 (why the indicator, not the mic meter)**: the mic level meter is NOT a valid confirmation signal for the global shortcut — the M5 capture stream runs continuously, so the meter responds to voice at all times regardless of PTT state. The PTT-active indicator is driven by the PTT press/release events and reflects shortcut firing directly (see T023/A1). See FR-011 for scope — fullscreen DirectX mode on Windows is explicitly out of scope. (B2 — test note: this scenario requires iRacing in borderless-windowed mode so both the settings page and sim window are simultaneously visible, or a secondary screen for observing the indicator while the sim has focus. The test cannot be performed in fullscreen mode.)
3. **Given** no PTT key has ever been configured (never-set state, distinct from a failed binding), **Then** a soft dismissable prompt is shown: "No PTT key set — click 'Set PTT Key' to bind one." (I3 — string aligned with FR-012/A2 and T023/F4; an earlier draft read "to enable voice queries.") This is NOT the FR-012 persistent error banner — that banner is reserved for cases where a binding *failed* (accessibility denied, key conflict). Never-configured and failed-to-register are two distinct states with different affordances: one is a first-time setup prompt, the other is a non-dismissable error until the condition is resolved.
4. **Given** the driver presses a key already reserved by the OS, **Then** the app informs them the binding could not be registered with a specific message (e.g., "This key is already bound by the OS — try a different key") and the driver may try again.

### User Story 4 — Engineer Personality Configuration (Priority: P2)

A driver tunes how the engineer communicates — how open, chatty, warm, assertive, and conscientious the voice feels — using sliders in the settings UI. Changes are reflected in the next engineer message without restarting the service.

**Why this priority**: Personality configuration is a key M5 output. Currently only achievable via Redis or config files. Exposing it through the UI makes it drivable.

**Independent Test**: Can be tested by moving sliders, saving, then requesting a Tier 3 message and verifying the tone matches the selected settings (manual verification).

**Acceptance Scenarios**:

1. **Given** the personality panel is open, **When** the driver moves a trait slider, **Then** a word label below the slider describes the current level (e.g., "Quiet" / "Balanced" / "Chatty" for energy).
2. **Given** the driver saves updated personality settings, **When** the Racing Engineer delivers the next message, **Then** it reflects the new trait values.
3. **Given** the driver sets the Energy trait to its minimum, **Then** the setting description notes that Tier 2 and Tier 3 commentary will be suppressed (Quiet mode).
4. **Given** the settings page opens, **Then** all five personality sliders display their current saved values, not defaults.

### User Story 5 — Telemetry Debug Readout (Priority: P2)

A driver (or setup helper) opens the debug panel and sees live telemetry variable values, Redis stream lag, and connection status — without needing a separate diagnostic tool.

**Why this priority**: The debug readout is essential for troubleshooting a session-start failure or verifying the telemetry pipeline is healthy before a race.

**Independent Test**: Can be tested with an active iRacing session (or recorded session replay) by opening the debug panel and verifying that displayed values match the expected telemetry (e.g., lap number increments, fuel level decreases).

**Acceptance Scenarios**:

1. **Given** a live iRacing session, **When** the debug panel is open, **Then** selected telemetry variables update at least once per second.
2. **Given** no iRacing session is running, **Then** the debug panel shows a "no session" indicator rather than stale values.
3. **Given** the Redis connection is healthy, **Then** stream lag is displayed as a numeric value (milliseconds); if lag exceeds 500 ms (per FR-018), a visual warning is shown.
4. **Given** the hub connection is lost mid-session, **Then** the connection status indicator updates to "disconnected" within 10 seconds without requiring a manual refresh. (The 10-second window matches FR-016's required stale threshold — the probe fires at 1Hz but the hub is only considered "disconnected" after 10 seconds without a successful probe result.)
5. **Given** the app has just launched, **Then** the Whisper STT status shows a loading/pending state until the model finishes loading, at which point it transitions to "Ready"; if model loading fails, the status shows "Load failed." The tri-state wire type, display-string mapping, and initial neutral-state rule are defined canonically in FR-016 (Whisper status tri-state) — this scenario tests that definition and does not redefine it. See T028/A3 for the `hubConnected` analog.

### User Story 6 — Voice Profile Upload (Priority: P3)

A driver uploads a short audio clip of their own voice (or a preferred voice) to use as the engineer's TTS voice via voice cloning. The uploaded profile is used for subsequent TTS audio.

**Why this priority**: Voice cloning is a differentiating feature but not required for the engineer to function. It's the lowest-priority setting — the engineer speaks with a default voice until a profile is uploaded.

**Independent Test**: Can be tested by uploading a short MP3 clip, clicking "Apply Voice Profile," then triggering a test TTS phrase and listening to the output.

**Acceptance Scenarios**:

1. **Given** the driver selects an audio file and uploads it, **Then** a progress indicator is shown, and on completion, the file name and upload timestamp are displayed.
2. **Given** the upload completes successfully, **When** the driver clicks "Test Voice," **Then** a short sample phrase plays using the cloned voice.
3. **Given** the driver uploads a file that is too short or too long, **Then** the hub rejects it with a 422 (duration validation is hub-side only — FR-022's third gate; the client cannot pre-check duration), the app displays the hub's informative error message ("Audio must be between {min} and {max} seconds (got Xs)"), and no profile is stored. (I2 — an earlier draft said the file is "not submitted"; only FORMAT failures are blocked client-side before submission.)
4. **Given** no voice profile has been uploaded, **Then** the TTS service uses its default voice and the settings page shows "Default voice (no profile uploaded)."

### User Story 7 — Telemetry Logging Toggle (Priority: P3)

A driver enables raw session telemetry logging from the settings page. After a session, the logged data is available for later IBT-style analysis.

**Why this priority**: Telemetry logging is an opt-in feature for future post-session analysis. It does not affect real-time operation and has no impact until the analysis tooling exists.

**Independent Test**: Can be tested by enabling the toggle, starting a session (or simulation), then confirming a log file appears in the expected output directory.

**Acceptance Scenarios**:

1. **Given** the toggle is switched on, **When** a new session starts, **Then** raw telemetry is written to a log file in a configured output directory.
2. **Given** the driver switches the toggle off mid-session, **Then** logging stops promptly: the logger drains all queued frames already in its buffer and closes the file cleanly, then accepts no new frames. Drain MUST complete within 500ms of receiving the stop signal (or within 100ms after the last frame was enqueued if the channel is already empty, whichever is shorter) — this bound makes the acceptance criterion testable. This preserves file integrity (no partial frame writes) without waiting for the session to end. See T031 for the implementation contract.
3. **Given** the output directory runs out of disk space, **Then** logging stops and a non-blocking warning is shown in the Logging tab (F1 — an earlier draft said "debug panel"; the `telemetry:log-warning` banner is wired in the Logging tab per T032, where the logging toggle lives); the real-time data path is unaffected.

### Edge Cases

- What happens when all saved audio devices are unavailable at app start (e.g., USB hub disconnected)? → Resolved by T014: each unavailable device emits `audio:device-unavailable` and falls back to the system default independently. If both input and output are unavailable simultaneously, two events are emitted and a combined warning is shown in the Audio tab ("Both saved audio devices are unavailable — please reselect your microphone and speaker" — F3, canonical string per T014/C4). Audio subsystem continues with system defaults. **U1 (late-mounting tab)**: Tauri events are not replayed to late subscribers — the Audio tab mounts long after startup and would miss init-time events. T014 therefore ALSO records unavailable-device status in managed state, queryable on tab mount (see T014/U1, T015/U1); the live event covers mid-session unplug only.
- What if audio device enumeration returns an empty list entirely (e.g., headless environment or VM with no audio hardware)? → The app MUST disable the affected audio subsystem gracefully, emit a structured warn log, and display the per-type message in the Audio tab ("No audio input devices found" / "No audio output devices found" — the canonical FR-002/T015-U1 strings; I7) without crashing (addressed by FR-002/FR-003, T014, and T015/U1).
- How does the app handle a hub server URL that resolves but the hub service is not running (TCP connect OK, HTTP fail)? → Resolved by T020: `check_hub()` distinguishes "TCP connection refused" (port not open) from "TCP connect OK but HTTP error response" (hub process running but returning non-2xx). These produce different UI messages: "Connection refused" vs. "Hub returned HTTP {status}: {reason}". Verify this distinction exists in the M4 `check_hub()` implementation or add it as part of T020.
- What if a personality slider value saved in config is outside the valid 1–5 range (corrupted config)? → Resolved by FR-026 + T025: out-of-range trait values are clamped to 1–5 on load in `personality-config.ts`; `save_config()` never writes an out-of-range value. (Test ownership split per T025/F3-C1: clamp-on-load is asserted in the hub TypeScript suite; the save-side guard is asserted in the Rust `commands.rs` suite, since `save_config()` is a Rust command.)
- What happens to in-progress voice capture if the PTT binding is changed mid-session? → Resolved by T022: if `bind_ptt_hotkey()` is invoked while the PTT key is physically held down (capture active), the rebind is refused with `Err("ptt:capture-in-progress")` and the driver is prompted "PTT is currently active — release the key first, then rebind." (I4 — string aligned with T022/U2 → T023's mapping.) The active capture completes normally to the key-release event; the rebind does not interrupt it.
- What if the driver attempts to upload a non-MP3 file (e.g., WAV or OGG)? → Resolved by FR-022: non-MP3 files are rejected by magic byte check in `upload_voice_profile()` with a "File must be an MP3" error before upload begins; the hub endpoint also validates MIME type as a second gate.
- What if a Tier 3 LLM synthesis call is already in progress when the driver saves a new LLM model name? → The in-flight synthesis call MUST complete with the model that was resolved at its start — no mid-synthesis model switch. The hub reads `hub:config:llm` once per synthesis request at the start of that request; changes to the Redis key during an active call do not affect it. The new model takes effect on the next synthesis request. (E1 — tested by T021's no-restart assertion sequence.)
- What happens to unsaved changes in one tab when the driver switches to another tab and back? → Resolved by T006/T008: the parent Settings page owns a single lifted `formState` object; tabs are display-only views over it. Switching tabs MUST NOT reset or reinitialize `formState` — a value edited in the Audio tab is still present after visiting Connection and returning, until Save or app close. (B4 — verify manually during any quickstart scenario: edit a field, switch tabs, return, confirm the edit is retained before saving.)
- What if `app_data_dir()` resolution fails at startup (e.g., sandboxed or misconfigured environment)? → `telemetry_log_dir` remains the sentinel `""` (T001). The Logging tab detects the sentinel, shows "Log directory path could not be resolved" (T032/C1), and disables the logging toggle (FR-019/U3) — logging cannot be enabled until the path resolves. All other tabs remain fully functional; the failure is contained to the Logging tab. (C2)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The app MUST provide a settings panel accessible from the main navigation without starting a session.
- **FR-002**: The settings panel MUST enumerate all available system audio input devices and allow the driver to designate one as the PTT microphone source. If enumeration returns an empty list (no audio hardware present), the app MUST display a "No audio input devices found" message and disable the input selection (see Edge Cases and T014). (B1 — device scope: All devices returned by the OS audio device enumeration API are listed without filtering — virtual audio devices such as VoiceMeeter or BlackHole appear as regular entries; it is the driver's responsibility to select the correct device.)
- **FR-003**: The settings panel MUST enumerate all available system audio output devices and allow the driver to designate one as the engineer voice output. If enumeration returns an empty list, the app MUST display "No audio output devices found" and disable the output selection (B1 — mirrors FR-002's empty-enumeration clause; behavior detailed in Edge Cases and T015/C2-U1).
- **FR-004** *(traceability alias for FR-025)*: Audio device selections MUST persist across app restarts. This requirement is a display alias — a specific application of FR-025 to audio device configuration, retained as a separate ID only for task-assignment and traceability clarity. It defines NO new behavior and requires NO separate implementation: do not implement audio persistence twice. T015's `save_config()` call satisfies FR-004 and FR-025 simultaneously. (T039 handles save-failure surface display, not audio persistence, and is not a co-implementer of FR-004.)
- **FR-005**: A microphone level meter MUST be visible when an input device is selected, updating in real time to confirm the device is active. The meter MUST update at a minimum rate of 10Hz (~100ms intervals) — sufficient for visible responsiveness to voice input. (A1 — rate is an implementation target defined in T012/E2; the 10Hz floor is the spec-level requirement.)
- **FR-006**: An audio output test button MUST play a short sample through the currently selected output device without starting a session.
- **FR-007**: The settings panel MUST provide editable fields for: Redis URL, hub server URL, LLM API base URL, LLM model name, and LLM API key (displayed as a masked password field; may be left empty for local endpoints that do not require authentication). **Note (E4)**: The `connection_token` field in `AppConfig` (existing M4 field) is intentionally not exposed in the Connection tab in M10 — it is unused in the current milestone and hidden from the UI. It remains in `AppConfig` for backward compatibility. **Note (C1 — default value UX)**: The LLM API base URL and model name fields are pre-filled with developer-specific homelab defaults (`https://lemonade.tdkottke.com/v1`, `user.Ornith-1.0-35B-GGUF`). On first launch, the UI SHOULD display a one-time hint adjacent to these fields (e.g., a muted annotation: "Default — update for your setup") so a first-time user understands the pre-filled values are not universally applicable. The hint reappears on every launch until the first successful Save sets `first_launch_seen = true` in `AppConfig` — a failed first Save does NOT suppress the hint (the driver must successfully save at least once to dismiss it permanently). **U1 clarification**: The hint is suppressed after any first successful Save — whether or not the LLM fields were modified from their defaults. The hint's purpose is to notify the driver on first launch that defaults exist, not to force modification; saving the defaults unchanged is a valid way to dismiss the hint. **A2 scope note**: "Save" here means the explicit settings-form Save button. The internal PTT auto-save performed by `bind_ptt_hotkey()` (T022/C5) does NOT set `first_launch_seen` and does NOT suppress the hint — a driver who binds PTT before ever visiting the Connection tab keeps the hint.
- **FR-008**: Each connection service MUST have an associated "Test" action that reports success or a specific failure reason. Tests are per-service (not per-field): one "Test" button for Redis, one for hub server, one for the LLM endpoint (covering base URL + model + API key together). (D3 — non-synthesis constraint: The LLM test MUST probe only connectivity and auth via `GET {base_url}/models` — NOT a real synthesis call. **A1 boundary rule**: any call to a completions/inference endpoint counts as synthesis for this purpose regardless of token count — a "1-token completion" probe is NOT permitted. A synthesis call in this path would bypass the Constitution V observability gate.)
- **FR-009**: A non-empty LLM model name change MUST take effect for the next Tier 3 message without requiring an app restart or hub server restart. The LLM Model Name field MUST be non-empty; inline validation (FR-026) MUST block Save if the model name field is blank — an empty model name is not a valid configuration (the hub falls back to engineer-config.json defaults but this fallback is not the intended user-facing behavior). "Valid" means a non-empty string that passes save; the hub uses it as-is in the LLM API `model` field — reachability is confirmed separately via `check_llm()` (FR-008) but is not a pre-condition for the change taking effect. **Scope (C6/A2 — which fields the no-restart guarantee covers)**: The no-restart guarantee covers the LLM model name AND the LLM base URL — both propagate via the `hub:config:llm` Redis key, which the hub reads at the start of every synthesis request (T018), so both take effect on the next Tier 3 message. Redis URL and hub server URL changes are saved to `AppConfig` immediately but take effect only after an app restart — those connections are established at client startup and have no live-reconnect path in M10. This scope definition is authoritative; SC-002 measures it and does not extend it.
- **FR-010**: The settings panel MUST provide a PTT key binding control that captures the next keypress and stores it as the global PTT shortcut.
- **FR-011**: The PTT binding MUST work as a global (OS-level) shortcut in windowed and borderless-windowed display modes, firing even when the sim window is in focus. Fullscreen DirectX mode on Windows is out of scope — global shortcuts may not fire reliably in that mode due to OS input routing (see Assumptions); Stream Deck passthrough is the recommended workaround for that scenario.
- **FR-012**: If the PTT binding cannot be registered (e.g., OS accessibility permission denied or key conflict), the app MUST display a specific error message and indicate that voice queries are disabled. The affordance MUST be a persistent warning banner in the Hotkeys tab reading "Voice queries disabled — PTT key could not be registered: [specific error]." This banner persists until the binding is successfully re-registered; it does not auto-dismiss. **A1 (banner-triggering code partition)**: Not every `Err` from `bind_ptt_hotkey()` triggers this banner. The persistent banner applies ONLY to registration failures that leave NO working PTT binding: `ptt:key-conflict` or `ptt:accessibility-denied` when no previous binding was restored (first-ever bind failed, or the restore also failed — e.g., accessibility revocation breaks re-registration of the old key too). Transient interaction outcomes — `ptt:timeout` (no key pressed within 10s) and `ptt:capture-in-progress` (key held when rebind attempted) — show inline transient messages only, NEVER the banner: an aborted capture flow does not disable voice queries. If a REBIND fails but the previous working binding was restored (T024 asserts restoration on failure; a `ptt:key-conflict` on the new key leaves the old key re-registerable), voice queries still work — show a dismissable warning "Couldn't bind the new key: [specific error] — keeping [previous key]" instead; the "Voice queries disabled" banner text would be factually wrong in that state. **A2 (never-configured vs. failed-binding distinction)**: The FR-012 persistent banner applies ONLY to failed binding attempts within the A1 subset above. The never-configured state (`ptt_hotkey == ""`) uses a distinct soft dismissable prompt ("No PTT key set — click 'Set PTT Key' to bind one") that is NOT a banner and IS dismissable. (**I1 note — default change required**: M5's code default for `ptt_hotkey` is `"F13"` (state.rs), under which this never-configured state would be unreachable; T001/I1 changes the `AppConfig` default to `""` in M10 so fresh installs actually enter this state. Configs upgraded from M5 retain their explicitly saved key. Startup PTT registration skips `""` — never-configured is a valid state, not a registration failure.) Developers reading only FR-012 MUST NOT apply the persistent-non-dismissable pattern to the never-configured first-run state. See US3 scenario 3 and T023/F4 for the implementation split.
- **FR-013**: The settings panel MUST expose five personality trait sliders (openness, warmth, energy, conscientiousness, assertiveness), each with a 1–5 range and human-readable level labels.
- **FR-014**: When Energy is set to 1, the UI MUST indicate that Tier 2 and Tier 3 commentary will be suppressed.
- **FR-015**: Personality settings MUST be saved to the hub server's configuration store (via Redis `hub:config:personality` key — JSON value: `{ openness, warmth, energy, conscientiousness, assertiveness }`, each an integer 1–5; see `data-model.md` for the full schema) when the driver clicks "Save," without requiring a hub restart.
- **FR-016**: The settings panel MUST include a telemetry debug panel showing: current iRacing session status, a fixed set of telemetry variables (fuel remaining, current lap number, track position, current lap time delta), Redis stream connection status and lag in milliseconds, hub server connection status (derived from a non-blocking periodic health probe that fires at 1Hz ± 200ms alongside the debug snapshot emission loop — the probe result is NOT awaited on the snapshot critical path; instead a `last_hub_probe_ok` timestamp is cached and hub is considered "connected" if the cached timestamp is within 10 seconds, "disconnected" otherwise; the 10-second stale threshold is a hard constraint; the 1Hz probe rate is the nominal target — rate degradation under the single-flight guard's backpressure (T028/A1b) is compliant, per the B2 clarification that follows (A4 — these two statements are reconciled, not contradictory). **B2 clarification**: The "1Hz ± 200ms" rate is the nominal rate under normal conditions. Under the single-flight guard (T028/A1b) with a 2-second probe timeout, the effective rate may drop well below 0.8Hz if the hub is slow or unresponsive — this is acceptable per T028/A1b design and does NOT constitute a violation of FR-016. Acceptance testing of US5 scenario 4 ("disconnected within 10 seconds") measures from the last successful probe, not from the start of hub unavailability), and Whisper STT model load status. **Whisper status tri-state (B1 — canonical definition)**: the Whisper STT status field has wire type `boolean | null` (Rust `Option<bool>`): `null` = loading/pending (model load not yet complete, or no status available yet), `true` = loaded successfully, `false` = load failed. The frontend maps these to the display strings "Loading…", "Ready", and "Load failed" respectively. Until the first status value is available, the UI MUST show the neutral loading state — never "failed" or "disconnected". This definition is canonical — US5 scenario 5, `data-model.md` `DebugSnapshot`, and T028/C2 defer to it rather than redefining it; if the M5 Whisper init path cannot signal failure separately from non-completion, `null` covers both states and the limitation is documented in the PR (T028/C2 fallback). The variable set is predetermined and not user-configurable.
- **FR-017**: The debug panel MUST update its values at least once per second while a session is active and show a "no active session" state when not connected. **U4 (hub loss during session)**: If the hub becomes and remains disconnected for an extended period (>60 seconds) during an active iRacing session, the debug panel MUST continue emitting snapshots at the normal rate with `hubConnected: false` — it does NOT stop emitting due to hub loss alone. Only iRacing session end triggers the "no active session" state; hub connectivity is an independent status field within the snapshot.
- **FR-018**: If stream lag exceeds 500 ms, the debug panel MUST display a visible warning.
- **FR-019**: The settings panel MUST include a toggle for raw telemetry session logging, with a read-only path display showing where logs are written. The log directory path is derived from the platform app data directory (`app_data_dir()/logs/telemetry`), stored in `AppConfig.telemetry_log_dir` for future extensibility, and is not user-editable in this milestone — the UI displays the value but does not expose an input for it. **Note (C1 — sentinel resolution)**: `telemetry_log_dir` defaults to `""` in `AppConfig::default()` (a sentinel — `app_data_dir()` cannot be called without a Tauri handle); it is resolved once at startup in the init flow and stored in the running `AppConfig`. Subsequent reads of the field always return the resolved path, never `""`. If the field is still `""` when the Logging tab reads it (e.g., init flow failed), show an error state in the Logging tab rather than displaying a blank path. **U3 (toggle disabled when path unresolved)**: If `telemetry_log_dir` is still the sentinel `""` when the Logging tab renders, the logging toggle MUST be disabled (greyed out, non-interactive) and the error state shown. The driver MUST NOT be able to enable logging until the path is resolved — an unresolved path means logs would have no valid destination. **Note (F3 — logged field set, I2-corrected)**: The log frame's exact field set is defined by `data-model.md` `TelemetryLogFrame`: ts, sessionId, fuel, lapDistPct (track position), lap, speed, gear, RPM, throttle, brake, lateral acceleration, longitudinal acceleration. This overlaps but does NOT superset the debug panel's 4 display fields: **lap time delta is intentionally not logged** — it is a derived value (vs. personal best) computed for the debug display, and the log captures raw telemetry only; derived values can be recomputed during analysis. The "fixed set" language in FR-016 refers to the debug display only.
- **FR-020**: Telemetry logging MUST NOT affect real-time telemetry ingestion latency or the Racing Engineer path (isolated write path).
- **FR-021**: The settings panel MUST provide a voice profile upload control that accepts MP3 files and submits the file to the hub server, which validates it and writes it to the Chatterbox reference audio directory via a Docker shared volume. (See FR-022 for format and duration validation requirements.)
- **FR-022**: The Tauri client MUST validate the MP3 format by magic byte check **(client-side — first gate)** before upload begins — wrong format is rejected immediately with "File must be an MP3" without making a network request. The hub server ALSO validates magic bytes **(hub-side — second gate)** after receiving the multipart payload, before writing to disk (see T034/I5). Duration validation is performed by the hub server after upload using `music-metadata` **(hub-side only — third gate)**; limits are stored as named constants `minVoiceProfileDurationSecs` and `maxVoiceProfileDurationSecs` in engineer-config.json (assumed 3–60 seconds — confirmed in T034b before implementation). If duration is out of range, the hub returns a 422 with "Audio must be between {min} and {max} seconds (got Xs)" and no file is written. **A3 clarification**: the "before upload begins" clause applies to format (magic bytes) only; the hub performs its own independent magic byte check and is NOT required to trust the client's result. **C1 (canonical magic byte list)**: The accepted MP3 magic byte sequences are defined ONCE in `data-model.md` ("MP3 magic byte sequences — shared reference"): `FF FB`, `FF F3`, `FF F2` (frame sync), `49 44 33` (ID3). FR-022, T034, T036, and T038 all defer to that list. If the accepted set ever changes, update `data-model.md` first and both gate implementations together — never edit an inline copy independently.
- **FR-023**: After successful voice profile upload, a "Test Voice" control MUST play a sample phrase using the cloned profile. The mechanism is the existing `test_audio_playback()` command — after upload, the hub's in-memory TTS client has already switched to the new reference file (updated by the upload endpoint, T034), so any audio playback via the hub naturally uses the cloned voice. **C3 (fresh synthesis, not a recorded clip)**: `test_audio_playback()` triggers a real-time Chatterbox TTS synthesis at click time using the M4 command's existing standard test phrase (unchanged by M10) — it does NOT replay a pre-recorded WAV and does NOT reuse the ephemeral `testClipUrl` from the upload response; verify during T037 that the M4 command path does reach `TtsClient` synthesis (T037/U4). Two distinct clips exist: (1) the upload-time confirmation clip synthesized by T034 ("Racing engineer online. Voice profile active."), returned as the ephemeral `testClipUrl`; and (2) the on-demand "Test Voice" playback via `test_audio_playback()`, reusable at any time after upload. US6 scenario 2 refers to mechanism (2). No separate dedicated TTS synthesis command needs to be built for the "Test Voice" button. **I1 (hub state dependency)**: This mechanism requires that the hub process remains running since upload — a hub restart after upload recovers the profile from `hub:config:voice-profile` in Redis (T035), so the Test Voice button continues to work after hub restarts as long as T035 startup recovery runs correctly. FR-023 does NOT guarantee test clip playback if neither the hub's in-memory state is set NOR the Redis key exists.
- **FR-024**: If no voice profile is uploaded, the app MUST display "Default voice (no profile uploaded)" (F3 — the canonical string per US6 scenario 4 and T036/T037; an earlier draft abbreviated it to "Default voice") and the engineer uses the TTS service's default voice.
- **FR-025**: All settings MUST be saveable and retrievable without data loss across app restarts.
- **FR-026**: Invalid field values (malformed URLs, out-of-range slider values) MUST be caught with inline validation before saving; the save action MUST be blocked until all fields are valid.
- **FR-027**: When a save operation fails (e.g., hub server unreachable, local disk error), the app MUST retain the unsaved changes in the form, display a specific error message identifying the cause, and allow the driver to retry without re-entering values. No partial save should leave settings in an inconsistent state. For hub sync specifically: `hub:config:personality` and `hub:config:llm` MUST be written together in a single Redis pipeline (not sequential awaits) so both SET commands are sent in one round-trip. If the pipeline fails, the entire hub sync is reported as failed and both keys are retried on the next save — neither is left in a partially-updated state. This pipeline-batched atomicity applies to the hub Redis sync path only; local disk save is independent and may succeed while hub sync fails. In that case the local save succeeds and a non-blocking hub sync warning is surfaced to the driver (see T039). If local disk save fails, hub sync MUST NOT be attempted (to prevent the hub from having newer settings than the local config). **Retry behavior**: When the driver explicitly retries (clicks Save again after a transient disk error is resolved), hub sync MUST be attempted if local disk save succeeds in that retry — the skip-hub-sync rule applies only when local save fails within the same Save invocation, not across retries. **U2 (startup resync after hub sync failure)**: If a Save call succeeds locally but hub sync fails (hub unreachable), the hub's Redis keys may be out of date when the hub reconnects. The Tauri client does NOT perform automatic startup resync in M10 — this gap is acceptable because the hub reads `hub:config:llm` and `hub:config:personality` at synthesis time (not at startup), so the next Save after reconnection will update Redis before the next synthesis. If the driver never saves again in that session, the hub uses stale values until the next explicit Save. Document this limitation in the PR description for T017.
- **FR-028**: On the first PTT binding attempt on macOS where Accessibility permission has not been granted, the app MUST display an instructional message explaining the requirement ("Accessibility permission required — open System Preferences → Privacy & Security → Accessibility") and MUST include a button that opens the Accessibility preferences pane directly. (C2 — upgraded from SHOULD; T023 treats this button as mandatory and this MUST alignment is correct.) This condition MUST NOT prevent the driver from using the rest of the app. **Windows behavior (U1)**: On Windows, if `register()` returns a permission-related error, display a generic message: "Could not register the PTT key — the OS rejected the binding. Try a different key or run the app as administrator." (F3 — canonical string aligned with T023/C5.) The macOS "Open Accessibility Settings" button MUST NOT be shown on Windows. See T023/C5 for the implementation mapping.
- **FR-029**: For every Tier 3 LLM synthesis event, the `engineer_events` audit table MUST record the LLM model name and base URL used in that call (Constitution III & V compliance — III: agent decisions auditable; V: every LLM interaction written to Postgres; F6 — both principles apply, cited together consistently). Enforcement: a unit test in `tier3-synthesizer.test.ts` that asserts `llm_model` and `llm_base_url` are present in every inserted `engineer_events` row MUST pass before merge, enforced by the strongest available mechanism in this order (I3 — the ladder from plan DoD F1/C2 and T042/C2): (a) CI workflow running the hub test suite — the preferred mechanism. **I1 correction (workflow contents verified 2026-07-08)**: `.github/workflows/ci.yml` EXISTS but currently runs only `cargo test -p iracing-engineer-lib` and `npm run build -w apps/hub-server` — NO step runs `npm test`, so the write-path test would never execute in CI as-is; T043's gate work therefore includes ADDING an `npm test -w apps/hub-server` step to `ci.yml` before any T017–T021 merge (an earlier revision of this clause wrongly claimed the workflow already ran `npm test`); (b) Husky pre-commit hook if CI is ever absent; (c) mandatory PR checklist item as a last resort, explicitly flagged as an enforcement gap with a follow-up ticket. See T043 and plan.md DoD F1.
- **FR-030**: The `/diagnostics` route MUST redirect to the Settings panel with the Debug tab active, preserving any existing in-app navigation links that reference that route. A blank page or 404 at `/diagnostics` is a regression. See T009.

### Key Entities

- **AudioDeviceConfig**: Selected input device identifier, selected output device identifier — persisted in client config store.
- **ConnectionConfig**: Redis URL, hub server URL, LLM base URL, LLM model name, LLM API key (optional, stored masked) — persisted in client config store; LLM base URL and model name propagated to hub at save time via Redis `hub:config:llm` key. API key stored locally only — never written to Redis.
- **PttBinding**: OS-level key/button identifier for push-to-talk — persisted in client config store; registered as a global shortcut on load.
- **PersonalityConfig**: Five trait values (1–5 each): openness, warmth, energy, conscientiousness, assertiveness — written to hub server on save.
- **VoiceProfile**: Uploaded audio file reference; submission status (uploading / active — "none" is represented by key absence, never stored); upload timestamp; duration — persisted in Redis `hub:config:voice-profile` ONLY, intentionally NOT in the client config store (I1 — see `data-model.md` and T037/E4: the Voice tab fetches state via `get_voice_profile()` on mount; adding AppConfig persistence for this entity is explicitly forbidden); audio data written to Chatterbox reference audio directory via Docker shared volume.
- **TelemetryLoggingConfig**: Enabled flag, log output directory path — persisted in client config store.

## Success Criteria *(mandatory)*

> **Numbering note**: These success criteria use SC-001 through SC-008. The `quickstart.md` validation scenarios also use SC-1 through SC-8 but are a separate numbering sequence for end-to-end test scenarios. To avoid ambiguity, spec criteria are always written as three-digit (SC-001) and quickstart scenarios as single-digit (SC-1).

### Measurable Outcomes

- **SC-001**: A driver with no prior config can complete first-time audio setup (device selection + test playback) in under 3 minutes. Measured from the moment the Audio tab first opens to when the test clip plays through the selected speaker (aligns with quickstart.md SC-1 timing definition).
- **SC-002**: LLM model name changes take effect immediately: after saving a model name change, the next Tier 3 message uses the new model within the same session, with no app restart or hub server restart. The authoritative requirement — including which fields the no-restart guarantee covers (LLM model name and LLM base URL via `hub:config:llm`; NOT Redis/hub URLs, which require an app restart) — is FR-009; this criterion measures FR-009 and does not restate or extend it.
- **SC-003**: A PTT binding can be recorded and confirmed functional in under 30 seconds. Confirmation signal: the PTT-active indicator in the Hotkeys tab (US3 scenario 2, T023/A1) — NOT the mic level meter, which is always-on and PTT-agnostic (A1).
- **SC-004**: All five personality traits are adjustable and confirmed saved in a single settings panel visit without navigating between pages.
- **SC-005**: The debug panel shows live telemetry lag within 1 second of an actual lag event (per FR-018: lag > 500 ms triggers the warning). **A4/B2 (appearance threshold + first-snapshot guard)**: The very first snapshot received after the Debug tab opens NEVER triggers the warning, regardless of its lag value — it only initializes the `hasReceivedFirstSnapshot` guard (T029/B2); the initial tab-open state shows no warning. From the second received snapshot onward, the warning appears immediately on any snapshot with lag > 500 ms — there is no appearance hysteresis beyond the first-snapshot guard. Only the dismissal path has the 5-second window. No false positives: the warning MUST NOT clear until lag has been continuously ≤ 500 ms for at least 5 wall-clock seconds (time-based hysteresis, not event-count-based). A single snapshot with lag > 500 ms within the 5-second dismissal window resets the timer — the full 5 seconds of continuous ≤ 500 ms lag must be observed again before the warning is dismissed.
- **SC-006**: A voice profile upload completes and is confirmed playable via the test phrase within 60 seconds for a 10–30 second audio clip.
- **SC-007**: Settings survive an app restart: all fields display their saved values on next open, with no fields reverting to defaults.
- **SC-008**: Real-time engineer path (Tier 1/2 alerts, telemetry ingestion) is unaffected by telemetry logging being enabled. **Measurable threshold (E3)**: P99 Tier 1 alert delivery latency MUST NOT increase by more than 50ms when the telemetry logging channel is at 80% capacity vs. logging disabled baseline. This threshold is verified by the manual quickstart SC-8 check (T040) — a subjective "no measurable increase" is not acceptable; if the 50ms bound cannot be held, the bounded-channel capacity (T030, default 1000 frames) MUST be tuned before merge.

## Assumptions

- Audio device enumeration uses the OS-native device list; virtual devices (e.g., VoiceMeeter) appear as regular devices.
- Personality configuration propagates via the Redis `hub:config:personality` key, written directly by the Tauri client's `save_config()` at save time and read by the hub (the existing M5 pattern) — no HTTP endpoint is involved. (I4 — an earlier draft of this assumption described an HTTP endpoint; FR-015, T017, and the plan all use the direct Redis write.)
- The LLM model name change propagates by writing to the hub server config; the hub reads the new model for the next synthesis request without a restart (consistent with M5's Redis-based personality config pattern).
- Voice profile upload sends the audio file to the hub server (`POST /api/voice-profile`), which validates the file and writes it to the Chatterbox reference audio directory via a Docker shared volume. Chatterbox has no file upload API — the hub acts as a proxy and the shared volume is how Chatterbox discovers the new voice reference file. This feature does not introduce a new service.
- Minimum voice profile duration is assumed to be 3 seconds and maximum 60 seconds — typical voice cloning requirements for Chatterbox; this can be adjusted once the TTS service limits are confirmed.
- The Whisper STT model runs in-client (M5 decision); there is no separate STT endpoint to configure. The "Whisper STT endpoint" from the original roadmap description is superseded by the in-client model — only the model file path or download status is surfaced in the UI, not a remote URL.
- Telemetry logs are written as newline-delimited JSON (NDJSON, `.ndjson`) files to a platform-derived local directory (`app_data_dir()/logs/telemetry`); the path is not user-editable in this milestone. The format and tooling for reading them are out of scope for this milestone.
- Stream Deck passthrough works via the same global hotkey mechanism as keyboard PTT; no dedicated Stream Deck SDK integration is required.
- On Windows, global shortcuts may not fire reliably when a fullscreen DirectX application (such as iRacing in fullscreen mode) intercepts all keyboard input before the OS can route it. Stream Deck passthrough (where the Stream Deck software routes button presses as OS-level keypresses) is the recommended workaround for fullscreen Windows use. The app does not attempt to diagnose this condition — if PTT fails in fullscreen, the driver should use Stream Deck or switch iRacing to borderless windowed mode. **Clarification (I2 — Stream Deck + fullscreen)**: Whether Stream Deck buttons also fire in fullscreen DirectX mode depends on the Stream Deck software's OS-level input injection privileges — it is NOT guaranteed to work in fullscreen DirectX on all Windows configurations. The claim that "Stream Deck is a workaround for fullscreen" assumes the Stream Deck software has elevated input access; this should be validated during acceptance testing if targeting fullscreen Windows use cases.
- Settings are stored in the Tauri client's local config store (platform-appropriate app data directory); no cloud sync.
- **Known M10 limitation — cloud LLM API keys**: The LLM API key is entered in the Connection tab but is NOT forwarded to the hub server in M10 (it stays in the local config only; `hub:config:llm` intentionally excludes `apiKey`). A driver who configures a cloud endpoint (e.g., `api.anthropic.com`) that requires authentication will receive silent 401 failures on every Tier 3 synthesis call. The Connection tab MUST surface a visible warning adjacent to the LLM Base URL field when the URL matches a known cloud pattern (hostname contains `api.anthropic.com`, `api.openai.com`, `generativelanguage.googleapis.com`, or `openrouter.ai`): "API key is stored locally but not forwarded to the hub server — Tier 3 synthesis will fail if this endpoint requires authentication. Cloud API key forwarding is planned for a future milestone." The Lemonade homelab default requires no auth and is unaffected. (D4 — Constitution IV's MUST clause is satisfied: the LLM provider remains runtime-switchable; the key forwarding gap is a known M10 limitation. The separate Constitution IV deviation — "Claude API is the default" named in the constitution vs. Lemonade shipped as the default — is covered by the project-owner sign-off in plan.md DoD B1, C1-IV extension, 2026-07-08.)
- **Known security limitation (M10)**: The LLM API key is stored as a plain string in the Tauri app data JSON config file. This exposes the key to any process that can read the app data directory. OS keychain integration (e.g., `tauri-plugin-keychain` or platform keychain APIs) is out of scope for M10 — it is a known tradeoff accepted for simplicity. This should be addressed in a future milestone if API keys with significant access are configured. Constitution IV is not violated (key stays local), but this limitation should be tracked.
- The settings panel is a dedicated page within the existing Tauri UI, not a separate window. It is organized into tabbed sections: Audio, Connection, Hotkeys, Personality, Debug, Voice, and Logging. The driver navigates between tabs without losing unsaved changes in other tabs.
