# Data Model: Redis Telemetry Publishing

## ConnectionEvent

Published to `iracing:events:connection` on iRacing state changes and as a snapshot on Redis connect/reconnect.

```json
{ "status": "Connected", "ts": 1719619200000 }
```

| Field | Type | Notes |
|-------|------|-------|
| `status` | `"Connected" \| "Disconnected"` | Current iRacing connection state |
| `ts` | number | Unix epoch milliseconds |

---

## SessionEvent

Published to `iracing:events:session` when a session begins, changes type, or ends.

**Active session:**

```json
{
  "active": true,
  "track_name": "Watkins Glen Boot",
  "player_car_name": "BMW M4 GT3",
  "player_car_idx": 3,
  "session_type": "Race",
  "wall_clock_time": "14:32:07",
  "ts": 1719619200000
}
```

**No session:**

```json
{ "active": false, "ts": 1719619200000 }
```

> **`player_car_idx`** — use this value to index into all `CarIdx*` arrays in the telemetry streams.
> It comes from `DriverInfo.DriverCarIdx` (root-level) in the iRacing session YAML, which is more
> reliable than `Drivers[0].CarIdx` for multi-driver teams.

---

## LiveTelemetryFrame

Published to `iracing:telemetry:live` at ≥ 60 Hz during an active session.
Wire format: flattened Redis Streams key-value pairs with one key per SDK field plus `_ts`.

**Field categories (60 Hz):**

| Category | Fields |
|----------|--------|
| Vehicle dynamics | `Speed`, `RPM`, `Gear`, `Throttle`, `Brake`, `SteeringWheelAngle` |
| Acceleration | `LatAccel`, `LongAccel`, `VertAccel` |
| Velocity | `VelocityX`, `VelocityY`, `VelocityZ` |
| Orientation | `Yaw`, `Pitch`, `Roll`, `YawRate`, `PitchRate`, `RollRate` |
| Position | `LapDistPct`, `LapDist` |
| Session state | `SessionTime`, `SessionTick`, `PlayerCarIdx`, `SessionFlags` |
| On-track state | `IsOnTrack`, `IsInGarage`, `OnPitRoad` |
| Camera | `CamCarIdx`, `CamGroupNumber`, `CamCameraNumber` |
| All-car position | `CarIdxLapDistPct`, `CarIdxTrackSurface` |
| All-car dynamics | `CarIdxGear`, `CarIdxRPM`, `CarIdxSteer` |
| Weather | `AirTemp`, `TrackTemp`, `TrackWetness`, `WeatherDeclaredWet` |

Array fields are serialized as JSON arrays: `"[0.12, 0.45, ...]"`.

## SessionTelemetryFrame

Published to `iracing:telemetry:session` at ≥ 15 Hz (every 4th live tick).

**Field categories (15 Hz):**

| Category | Fields |
|----------|--------|
| Fuel | `FuelLevel`, `FuelLevelPct`, `FuelUsePerHour` |
| Coolant / oil | `WaterTemp`, `OilTemp`, `OilPress` |
| Tire temps LF | `LFtempCL`, `LFtempCM`, `LFtempCR` |
| Tire temps RF | `RFtempCL`, `RFtempCM`, `RFtempCR` |
| Tire temps LR | `LRtempCL`, `LRtempCM`, `LRtempCR` |
| Tire temps RR | `RRtempCL`, `RRtempCM`, `RRtempCR` |
| Tire wear LF | `LFwearL`, `LFwearM`, `LFwearR` |
| Tire wear RF | `RFwearL`, `RFwearM`, `RFwearR` |
| Tire wear LR | `LRwearL`, `LRwearM`, `LRwearR` |
| Tire wear RR | `RRwearL`, `RRwearM`, `RRwearR` |
| Lap timing | `LapCurrentLapTime`, `LapLastLapTime`, `LapBestLapTime`, `LapDeltaToBestLap`, `LapDeltaToOptimalLap` |
| Session remaining | `SessionTimeRemain`, `SessionLapsRemain` |
| Hero position | `PlayerCarClassPosition`, `PlayerCarPosition`, `IncidentCount` |
| All-car positions | `CarIdxPosition`, `CarIdxClassPosition` |
| All-car laps | `CarIdxLap`, `CarIdxLapCompleted` |
| All-car timing | `CarIdxLastLapTime`, `CarIdxBestLapTime`, `CarIdxEstTime`, `CarIdxF2Time` |
| All-car status | `CarIdxOnPitRoad`, `CarIdxTireCompound`, `CarIdxFastRepairsUsed` |
| Pit service | `PitSvFlags`, `PitOptRepairLeft`, `PitRepairLeft` |

## Classification Rule

A field not in the session-rate set is published on the live stream (default).

```
if field_name IN SESSION_RATE_FIELDS → iracing:telemetry:session (15 Hz)
else                                  → iracing:telemetry:live    (60 Hz)
```

## M5 — Racing Engineer: LLM + Push-to-Talk

The Racing Engineer adds an LLM reasoning tier (Tier 3) on top of the M4 rule
engine. Delivery uses **Model A**: one `PriorityMessageQueue` drains all tiers, so
a Tier 1 alert preempts still-pending Tier 3 sentences (T1 > T2 > T3, FR-015).

### Channels

| Channel | Direction | Payload | Purpose |
|---|---|---|---|
| `engineer:query` | Tauri → hub (pub/sub) | `EngineerQuery` | transcribed push-to-talk question |
| `voice:audio` | hub → Tauri (pub/sub) | `AudioClipRef` | one per synthesized sentence (Tier 1/2 carry `eventType`, Tier 3 carry `tier3Type`) |
| `hub:config:personality` | Tauri → hub (KV) | 5-trait JSON | OCEAN personality, read by the engineer |

### Tier 3 pipeline (`tier3-synthesizer.ts`)

`trigger → suppression (Energy=1) → deference check → context assembly (token
budget) → versioned prompt (system-base + personality + per-type) → audit
pre-write → streamed LLM (tool loop + timeout) → sentence split → enqueue Tier 3
→ dispatcher TTS + publish → audit finalize`. LLM unreachable ⇒ skip + log; a
driver-query gets a canned line; Tier 1/2 are never affected.

### `engineer_events` (Postgres audit — FR-022, SC-008)

One row per LLM interaction, written **before** acting on the response. Columns:
`id, session_id, tier3_type, prompt, response, tools_called, latency_ms, outcome,
created_at`. See `migrations/001_engineer_events.sql` and
`contracts/engineer-events-audit.md`.

### STT

Push-to-talk transcription is **in-client** (`whisper-rs` base.en, POC-0002),
published to `engineer:query`. The Whisper model (`ggml-base.en.bin`) is provided
via `apps/tauri-client/src-tauri/models/download.sh` or `WHISPER_MODEL_PATH`.
